# Private Archive Vault Design

Date: 2026-03-09  
Status: Validated design draft  
Project: ForgetMe

## Summary

`ForgetMe` 的第一阶段不是“数字分身”产品，而是一个 **本地优先、证据链优先、单用户私有的私人档案库**。

它的核心目标不是立刻模仿一个人说话，而是先把与人有关的资料安全、完整、可追溯地保存下来：聊天记录、图片、文档及其附件都要能够成批导入、保留原样、建立证据链、支持可回滚的处理与脱敏，并为后续的人物画像、关系图谱、独立 agent 打下可靠基础。

一句话定义：**先做可信归档，再做可信理解，最后做可信生成。**

## Product Positioning

### Recommended Route

第一版选择 **“证据金库型私人档案库”**，而不是“记忆搜索引擎型”或“数字分身型”。

推荐原因：

- 资料可信度、可追溯性和长期保存价值，是这个项目最不可替代的核心
- 一旦原件层没有做好，后面的画像、图谱和 agent 都会失去可靠基础
- 高敏数据治理、删除权、继承权、误判风险都要求系统先成为“档案库”，而不是“角色模拟器”

### Confirmed MVP Direction

- 资料类型：聊天记录 + 图片 + 文档
- 部署形态：混合模式
- 存储原则：原件本地优先，索引与备份可选上云
- 第一核心能力：原件归档
- 使用模型：单用户私有库

### Market Gap

本项目的独特点在于：

- **原件层与分析层严格分离**
- **以“人”为中心组织多模态证据**
- **允许后续为每个人构建独立 agent，但不把 agent 放在第一阶段**
- **既支持个人档案，也为未来群体关系图谱保留结构**

## Architecture

系统推荐拆成四层，避免一开始做成一个难以治理的“大一统智能库”。

### 1. Vault Layer

原件层只负责保存“进入系统时的真实材料”。

存储对象包括：

- 聊天导出包
- 图片原图
- 文档原件
- 附件与压缩包
- 导入时的原始目录结构信息

设计原则：

- 不修改原件
- 可校验完整性
- 可追溯来源
- 可按批次回滚处理结果

建议元数据：

- 文件哈希
- 导入时间
- 来源渠道
- 原始路径
- 文件类型与大小
- 批次 ID
- 提取出的时间戳候选

### 2. Processing Layer

处理层负责从原件中提取结构化和可检索信息，但不能替代原件。

派生数据包括：

- OCR 文本
- 聊天消息结构化结果
- 图片 EXIF
- 文档文本分段
- 去重结果
- 脱敏版本
- 关键词索引
- 向量索引

原则：任何处理结果都必须能回跳到原件。

### 3. People Layer

人物层不追求人格结论，而是先建立稳定的身份锚点。

建议字段：

- 姓名 / 昵称 / 别名
- 联系方式碎片
- 平台账号线索
- 头像候选
- 首次出现时间
- 最后出现时间
- 出现频率
- 可信度

### 4. Graph Layer

关系层记录人物、文件、事件之间的连接。

第一版不做复杂知识图谱，只需要支持：

- 某文件涉及哪些人
- 某人出现在哪些批次和时间段
- 某事件关联了哪些聊天、图片、文档
- 某条结论来自哪些证据

## Ingest Workflow

导入采用一条严格的归档流水线：

`接收 -> 冻结原件 -> 批次登记 -> 文件级去重 -> 内容级近似去重 -> 类型化解析 -> 脱敏派生 -> 建立关联 -> 审计落盘`

### Key Rules

- 任何智能处理必须发生在原件冻结之后
- 系统先接收整批资料，再做理解和分析
- 去重分为文件级和内容级两种，近似去重只提示、不自动删除
- 删除与合并操作必须可回滚

### Type-Specific Parsing

- 聊天记录：抽消息、参与者、时间线、附件关系
- 图片：抽 EXIF、OCR、视觉候选信息
- 文档：抽文本、段落、附件和嵌入关系

### Batch-Level Guarantees

为实现“一组不差”，每次导入都应生成批次报告：

- 总文件数
- 已冻结文件数
- 解析成功数
- 解析失败数
- 疑似重复数
- 需要人工确认数

## Privacy and Security Model

### Default Boundary

第一版采用 **“本地原件不可默认出境，云端只接触派生数据”**。

本地保存：

- 原始聊天
- 原始图片
- 原始文档
- 原始附件
- 原始目录结构
- 原件与脱敏件的映射关系

云端可选保存：

- 加密备份块
- 检索索引
- 脱敏后的摘要与标签
- 同步所需最小元数据

### Audit Model

第一版不做复杂角色系统，但必须审计四类动作：

- 导入
- 查看
- 导出
- 删除

### Deletion Model

删除至少分为两种：

- 逻辑删除：从索引、人物、关系中移除
- 物理销毁：清除原件、派生件、备份块和相关索引

### Desensitization

第一版优先支持规则型脱敏：

- 手机号
- 证件号
- 地址
- 车牌
- 学校 / 单位名
- 人脸区域

脱敏必须生成独立派生件，不能覆盖原件。

## MVP Scope

### In Scope

- 批量导入聊天记录、图片、文档
- 原件冻结与本地保存
- 批次审计
- 哈希去重与近似去重提示
- 基础检索
- 基础人物锚点抽取
- 轻量人物 / 文件 / 事件关联

### Out of Scope

- 语音克隆
- 拟人聊天
- 自动人格模仿
- 多人协作
- 复杂知识图谱推理
- 群体行为模拟

## Technical Direction

推荐先做 **桌面端本地优先架构**，而不是纯 Web。

推荐组合：

- `Electron` 或 `Tauri`：桌面应用壳
- `SQLite`：结构化元数据、批次、人物、关系、审计
- 本地对象存储目录：原件和派生件
- 后台任务队列：OCR、解析、索引、去重、脱敏
- 可选云同步服务：备份与索引同步，不作为主库

### Why Desktop-First

- 更符合高敏资料本地保存的原则
- 更容易接入文件系统和批量导入
- 更适合长期个人资料库场景

## Error Handling and Validation

### Failure Policy

失败时优先保证“不丢原件”。

导入失败必须以批次报告呈现，且允许部分成功：

- 原件已冻结，但解析失败
- 文件损坏，等待人工处理
- 文档受密码保护，等待人工处理
- 编码异常，进入待审状态
- 元数据缺失，但原件已保存

推荐引入 `quarantine` / `needs review` 状态，而不是直接丢弃异常输入。

### MVP Verification Baselines

- 导入后文件数不减少
- 原件哈希可复验
- 派生数据可回跳到原件
- 删除流程可审计

建议后续测试优先围绕：

- 导入测试
- 断点恢复测试
- 去重误伤测试
- 删除与导出测试

## Roadmap

### Phase 1: Trusted Archive

- 做好原件归档、批次管理、去重、审计和基础检索

### Phase 2: Structured Understanding

- 做人物时间线
- 做事件聚合
- 做关系图谱

### Phase 3: Generative Layer

- 为每个人建立独立 agent
- 在证据约束下生成摘要、建议、对话风格模拟
- 探索群体画像与群体记忆

## Design Principles

- 先保存，再理解，再生成
- 原件层与分析层严格分离
- 所有结论都应有证据回链
- 默认本地优先，云端能力可选
- 错误时不丢原件
- 第一版追求可信，不追求炫技

## Next Step Recommendation

如果进入实现阶段，建议先写一份实现计划，拆成以下顺序：

1. 搭建桌面壳与本地存储目录
2. 定义 SQLite 数据模型
3. 做批次导入与原件冻结
4. 接入聊天 / 图片 / 文档解析管线
5. 做批次报告、去重和基础检索
6. 再补人物锚点与轻量关系层
